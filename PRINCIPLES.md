## 1. 프로젝트 정체성

### 1.1 핵심 가치 제안
```
문제: "집 보러 갈 때 뭘 봐야 할지 모르겠어요 😢"
해결: "지금 여기서 임장 시작하기" [버튼 1개]
```

**30초 요약**:
- 30대 여성이 집 보러 갈 때 느끼는 "판단의 외로움" 해소
- GPS 감지 → YouTube 영상 자동 재생 → 전문가 음성 안내
- 실거주자를 위한 체크리스트 자동 생성

### 1.2 왜 이 앱인가?

**기존 솔루션의 문제**:
```
❌ 네이버/다음 부동산: 숫자와 사진만 → "그래서 어쩌라고?"
❌ 부동산 유튜브: 영상 찾기 어려움 → 보러 가서도 못 봄
❌ 부동산 앱들: 투자 중심 → 실거주자 외면
❌ ChatGPT/AI: 판단 요구 → 더 외로워짐
```

**우리의 해결책**:
```
✅ 버튼 1개로 시작
✅ 걸으면서 자동 재생
✅ 전문가가 "지금 왼쪽 보세요" 안내
✅ 기준 제시 → 비교 → 안심
```

### 1.3 왜 우리를 믿어야 하나?

**789회 임장 경험** (2018.05.25 ~ 현재)
```
숫자가 말해주는 것:
- 전국 50개 이상 도시 방문
- 약 20,000km 임장 (지구 반 바퀴)
- 5년 이상 꾸준히 기록
- 실제 발로 뛴 데이터

→ 책상이 아닌 현장에서 나온 인사이트
→ [전국 임장 기록 지도 보기](https://www.google.com/maps/d/u/0/edit?hl=ko&mid=1IFcLVWKtWarWDJ-kH1P_DfvwbeiPO4Q&ll=37.520515003409415%2C127.00594821136168&z=13)
```

**경험의 깊이**:
- 임장연구소 운영
- YouTube, Instagram, TikTok 운영
- 도시 고고학자 김시덕님과 협업
- TV 프로그램 출연
- 전자책 출간: "부동산 초보를 위한 기초 및 강동구 임장"

**투명성**:
- 모든 임장 기록 공개 (구글 지도)
- 실제 영상 공개 (YouTube)
- 실명 활동

## 2. 3대 절대 원칙

### 원칙 1: 설명하지 마라, 느끼게 하라

**문제**: 설명은 불안을 증폭시킨다
```
❌ 나쁜 예:
"임장이란 부동산 현장을 직접 방문해 주변 환경, 교통, 
생활 편의시설 등을 체크하는 것입니다. 지하철역과의 
거리, 학군, 상권 등을 꼼꼼히..."

→ 사용자: "아... 그래서 뭘 해야 하는데?" 😰
```
```
✅ 좋은 예:
[큰 버튼 1개]
"지금 여기서 임장 시작하기"

→ 사용자: 탭! → 영상 재생 → "아, 이거구나!" 😊
```

**적용 사례**:

1. **홈 화면**:
   - 설명 없음
   - 버튼 1개만
   - 부가 설명: "켜놓고 걸으면 전문가가 속삭여요" (1줄)

2. **플레이어 화면**:
   - 영상 자동 재생
   - 체크리스트만 표시
   - "전문가가 안내해드려요" (짧게)

3. **체크리스트**:
   - 설명 없음
   - 기준 vs 실제
   - ✓ 또는 ?

**측정 방법**:
- 첫 사용 15초 내 가치 이해 >80%
- "이게 뭐예요?" 질문 <20%

---

### 원칙 2: 판단을 요구하지 마라, 안심시켜라

**문제**: 판단 요구는 외로움을 가중시킨다
```
❌ 나쁜 예:
"지하철역까지 거리가 적절한가요?"
[매우 적절 / 적절 / 보통 / 부적절 / 매우 부적절]

→ 사용자: "적절이 뭔데? 10분이 적절한 건가?" 😰
```
```
✅ 좋은 예:
✓ 지하철역 거리
  기준: 도보 10분 이내
  실제: 성수역 도보 7분
  
→ 사용자: "아, 기준보다 가깝네. 괜찮구나!" 😊
```

**적용 사례**:

1. **체크리스트 구조**:
```json
{
  "criterion": "도보 10분 이내",  // 기준 제시
  "result": "성수역 도보 7분",    // 실제 측정
  "status": "pass"               // 판단 대신 제공
}
```

2. **색상 시스템**:
   - 청록색 ✓ = 기준 충족 = 안심
   - 빨강색 ? = 확인 필요 = 주의
   - 판단은 사용자가 아닌 전문가가

3. **음성 안내**:
   - "이 정도면 괜찮아요" (O)
   - "어떻게 생각하세요?" (X)

**측정 방법**:
- 체크리스트 이해도 >90%
- "이거 어떻게 판단해요?" 질문 <10%

---

### 원칙 3: 흐름을 깨지 마라, 유지하라

**문제**: 중단은 몰입을 파괴한다
```
❌ 나쁜 예:
[영상 일시정지]
"주변 환경을 평가해주세요"
[별점 선택 화면]
→ 걸음 멈춤 → 몰입 깨짐 → 포기
```
```
✅ 좋은 예:
[영상 계속 재생]
"지금 왼쪽 보세요... 저 건물이..."
[체크리스트 자동 업데이트]
→ 걸음 유지 → 몰입 유지 → 완주
```

**적용 사례**:

1. **자동 재생**:
   - GPS 감지 시 즉시 재생
   - 일시정지 없음
   - 끝나면 다음 영상 자동 전환

2. **Hands-free**:
   - 화면 터치 불필요
   - 걸으면서 청취 가능
   - 음성만으로 완성

3. **체크리스트**:
   - 자동 생성
   - 입력 요구 없음
   - 나중에 확인 가능

**측정 방법**:
- 영상 완주율 >70%
- 중도 이탈률 <30%
- 재듣기율 >50%

## 4. 문제 정의

### 4.1 핵심 문제
```
"집 보러 갈 때 뭘 봐야 할지 모르겠어요"

깊이:
1단계: 정보 부족 (What)
2단계: 기준 불명 (How)
3단계: 판단 불안 (Why)
4단계: 외로움 (Who)

→ 근본 문제: "판단의 외로움"
```

### 4.2 문제의 크기

**시장 규모**:
- 30대 여성: 약 400만명
- 6개월 내 이사 예정: 약 40만명 (10%)
- TAM (Total): 40만명
- SAM (Serviceable): 20만명 (서울/수도권)
- SOM (Obtainable): 2만명 (10%, 1년)

**문제 심각도**:
- 집 보러 다니는 횟수: 평균 5-10회
- 1회당 시간: 2-3시간
- 총 소요 시간: 10-30시간
- 정신적 스트레스: 매우 높음
- 후회 확률: 30-40%

### 4.3 현재 해결책의 한계

**1. 네이버/다음 부동산**:
```
장점: 정보 많음
단점: 
- 숫자만 나열
- "그래서?" 해결 안 됨
- 현장 연결 약함
```

**2. YouTube 부동산 채널**:
```
장점: 전문가 의견
단점:
- 영상 찾기 어려움
- 현장에서 못 봄
- 일반론만 (구체적 X)
```

**3. 부동산 앱들**:
```
장점: 편리함
단점:
- 투자 중심
- 실거주자 외면
- 숫자 나열
```

**4. ChatGPT/AI 챗봇**:
```
장점: 즉답 가능
단점:
- 판단 요구 (더 외로워짐)
- 현장 정보 없음
- 책임 회피
```

**5. 친구/부모 조언**:
```
장점: 신뢰
단점:
- 시간 맞추기 어려움
- 전문성 부족
- 미안함
```

### 4.4 우리의 차별점
```
기존: 정보 → 사용자 판단 → 불안
우리: 정보 + 기준 + 판단 → 안심

기존: "여기 어때요?" (판단 요구)
우리: "여기는 이래서 괜찮아요" (안심 제공)

기존: 영상 따로, 현장 따로
우리: GPS로 자동 연결

기존: 투자 관점
우리: 실거주 관점
```

**Action (행동)**:
```
"지금 여기서 임장 시작하기" 버튼 클릭

→ 가장 쉬운 행동
→ 2초 내 완료
```

**Reward (보상)**:
```
가변적 보상:
- "아, 이거 봐야 하는구나!" (인지적)
- "기준보다 가깝네!" (안심)
- "뭘 놓칠 뻔 했어" (회피)

→ 예측 불가능한 인사이트
```

**Investment (투자)**:
```
체크리스트 쌓임:
- 본 집마다 기록
- 비교 가능
- 다시 보고 싶어짐

→ 재방문 유도
```

### 10.2 행동경제학

**손실 회피 (Loss Aversion)**:
```
문제: "뭘 놓쳤을까?" 불안
해결: 체크리스트로 "다 봤어" 확인

→ 후회 방지
```

**앵커링 (Anchoring)**:
```
기준 제시 → 비교 → 판단

예: "도보 10분 이내" (앵커)
    "성수역 도보 7분" (실제)
    → "괜찮네!" (판단)
```

**선택 과부하 (Choice Overload)**:
```
문제: 너무 많은 선택지 → 결정 마비
해결: 버튼 1개만

→ 빠른 시작
```

### 10.3 UX 원칙

**피크-엔드 규칙 (Peak-End Rule)**:
```
기억되는 것:
- 가장 강렬한 순간 (Peak)
- 마지막 순간 (End)

적용:

→ 긍정적 기억
```

**인지 부하 최소화**:
```
원칙: 생각하지 않아도 사용 가능

- 버튼 1개
- 자동 재생
- 입력 없음

→ 피로 감소
```

**신뢰 구축**:

## 12. 자동화 시스템 (신규)

### 12.1 JSON 기반 자동화의 의미

**"코드 수정 0줄로 새 코스 추가"**

**이전 (하드코딩)**:
```dart
// youtube_player_screen.dart
_buildChecklistItem('지하철역 거리', '성수역 도보 7분', true)
_buildChecklistItem('대형마트', '이마트 성수점 10분', true)
_buildChecklistItem('학교', '서울숲초등학교 확인 필요', false)

// home_screen.dart
if (location == '성수동') {
  videoId = 'hNHiVGzbcaI';
  locationName = '성수동';
}

→ 새 코스 추가 = 3개 파일 수정
→ 30분 소요
→ 개발자 필요
→ 실수 가능성 높음
```

**지금 (JSON)**:
```json
// assets/courses/seongsu.json
{
  "id": "seongsu",
  "name": "성수동",
  "media": {
    "videoId": "hNHiVGzbcaI"
  },
  "checklist": [
    {"title": "지하철역 거리", "result": "성수역 도보 7분", "status": "pass"}
  ]
}

→ 새 코스 추가 = JSON 파일 1개 생성 + ID 추가 (1줄)
→ 5분 소요
→ 비개발자도 가능
→ 실수 최소화
```

### 12.2 작동 원리

**1. 앱 시작 시**:
```dart
// 1. CourseService 초기화
CourseService().loadAllCourses()

// 2. JSON 파일들 자동 로드
for (courseId in ['seongsu', 'gangnam', ...]) {
  final jsonString = await rootBundle.loadString(
    'assets/courses/$courseId.json'
  );
  
  // 3. JSON → Course 객체 변환
  final course = Course.fromJsonString(jsonString);
  
  // 4. 메모리에 저장
  _courses[course.id] = course;
}
```

**2. 버튼 클릭 시**:
```dart
// 1. 코스 가져오기
final course = CourseService().getCourseById('seongsu');

// 2. 화면으로 전달
Navigator.push(
  context,
  MaterialPageRoute(
    builder: (context) => YouTubePlayerScreen(
      videoId: course.media.videoId,     // JSON에서!
      locationName: course.name,          // JSON에서!
      course: course,                     // 전체 객체!
    ),
  ),
);
```

**3. 플레이어 화면**:
```dart
// 체크리스트 동적 생성
...course.checklist.map((item) => 
  _buildChecklistItem(item)
).toList()

→ JSON만 바꾸면 화면 자동 변경!
```

- 최소 기능 제품
